# Node A 下一阶段推进计划

适用日期：`2026-04-01`

## 一、当前阶段判断

截至当前，`Node A + 雷达 + 云台` 的核心功能已经闭环验证完成。

当前不再建议优先新增主链功能，而建议进入“稳定性复测 + 集成前收口”阶段。

## 二、下一阶段目标

下一阶段建议按下面顺序推进：

1. 压稳现有硬件组合
2. 固化测试证据与验收标准
3. 再准备云端或多节点集成

## 三、本周优先任务

### 1. 稳定性复测

目标：

- 确认当前成功不是偶发
- 确认不同目标进入方向都能稳定复现

建议测试项：

- 左侧进入视场
- 右侧进入视场
- 正前进入视场
- 目标停留 3 到 5 秒
- 目标进入后离场
- 断电重启后重新跑一轮

建议每轮保留：

- 一次 `SUMMARY`
- 一段 `UPLINK,EVENT`
- 一段关键状态切换日志

### 2. 供电与接线鲁棒性验证

目标：

- 确认当前链路不是“摆在桌上刚好能用”

建议观察：

- 舵机大动作时是否重启
- 串口是否掉线
- 线束轻微扰动时是否丢目标
- 长时间上电后是否重新出现抖动

### 3. 结果固化

目标：

- 把“已跑通”变成后续可引用的阶段结论

建议保留：

- 一份最终 `SUMMARY`
- 一段真实雷达日志
- 一段演示视频或照片

## 四、当前不建议做的事情

在没有新问题暴露前，不建议优先做这些：

- 大改 `HunterAction`
- 大改 `GimbalController`
- 重命名现有协议字段
- 新增过多串口命令
- 再接入新硬件扩展主链

原因：

- 当前主链已通
- 现在最值钱的是稳定性和可复现性
- 过早改主链会破坏已经验证通过的结果

## 五、下一阶段代码侧建议

如果后面继续写代码，建议按下面优先级推进：

1. 日志收口与显示优化
   比如减少串口粘连、优化关键行可读性。
2. 测试辅助能力
   比如更清晰的最终测试快照，而不是继续扩主链逻辑。
3. 集成接口准备
   比如把当前 `UPLINK` 输出进一步对齐云端接入格式。

当前不建议优先继续扩展“识别逻辑”或“控制策略”。

## 六、阶段验收标准

下一阶段完成时，建议满足以下标准：

- 连续多轮真实雷达测试均可复现
- `track_active / track_confirmed / gimbal_tracking / track_lost / gimbal_lost` 可稳定出现
- `event_id` 可稳定生成
- 舵机不再出现异常抖动
- 重启后仍可复现相同链路

## 七、建议的下一步动作

如果继续推进，最合理的顺序是：

1. 再做 1 到 2 轮干净的真实雷达复测
2. 固定一份最终演示日志和视频
3. 再进入云端接入或多节点协同阶段
