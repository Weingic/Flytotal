## 2. 风险升级触发矩阵 V1
| 场景 | 输入动作 | 预期主链 | 预期事件 | 预期视觉/抓拍 |
|---|---|---|---|---|
| S1 无 RID 短时进入 | 短时 `TRACK` + `RID,MISSING` | `DETECTING/TRACKING` 或过渡态，不直接高风险挂死 | 不应直接 `OPEN` | 可无视觉锁定 |
| S2 持续无身份停留 | 连续 `TRACK` + `RID,MISSING` | 升到 `SUSPICIOUS/HIGH_RISK` | 允许进入 `OPEN` | 进入视觉准备态 |
| S3 合法目标进入 | 连续 `TRACK` + `RID,OK` | `NONE/NORMAL` 或低风险稳定 | 不应新开高风险事件 | 抓拍关闭 |
| S4 身份异常目标 | 连续 `TRACK` + `RID,SUSPICIOUS` | 快速升风险 | 可开启事件 | 视觉进入风险联动 |
| S5 高风险后视觉触发 | 高风险阶段维持目标 | 高风险保持 | `OPEN` 持续 | `vision_locked=1` 或 `capture_ready=1` |
| S6 目标丢失回落 | `TRACK,CLEAR` | 回落到 `LOST/IDLE` | `CLOSED` 且 `TRACK_LOST` | 退出锁定，抓拍结束 |

## 3. 状态报文格式 V1
建议状态报文类型：`UPLINK,HB`（心跳）/ `UPLINK,STATUS`（状态快照）。

核心字段（网页与联调都要对齐）：

1. 节点信息：`node`, `zone`, `role`, `timestamp`
2. 主链状态：`main_state`, `hunter_state`, `gimbal_state`, `rid_status`
3. 轨迹信息：`track_id`, `track_active`, `track_confirmed`, `x`, `y`, `vx`, `vy`
4. 风险信息：`risk_score`, `risk_level`, `risk_reasons`, `pending_risk_state`, `risk_transition_mode`
5. 视觉联动：`vision_state`, `vision_locked`, `capture_ready`
6. 事件摘要：`event_active`, `current_event_id`
7. 一致性字段：`consistency_status`, `consistency_expected_main_state`

判定要点：

- 状态报文用于“当前是什么状态”。  
- 网页显示与桥接一致性核对默认都以状态报文字段为准。

## 4. 事件报文格式 V1
建议事件报文类型：`UPLINK,EVENT`。

核心字段（事件留痕必备）：

1. 事件标识：`event_id`, `event_status`, `event_level`
2. 触发语义：`reason`, `event_close_reason`, `event_trigger_reasons`
3. 关联上下文：`track`, `hunter`, `gimbal`, `rid`, `main_state`, `risk_level`, `risk`
4. 位置与运动：`x`, `y`, `vx`, `vy`
5. 视觉/抓拍：`vision_state`, `vision_locked`, `capture_ready`
6. 时间与来源：`ts`, `node`, `zone`

建议新增（若后续上云需要）：

- `capture_path`：抓拍文件路径（本地或对象存储路径）  
- `capture_id`：抓拍记录唯一 ID  
- `evidence_bucket`：证据分组 ID（便于答辩展示）

## 10. 4.10 结果文件判定顺序（计划内）
建议固定按下面顺序看，避免来回跳文件：

1. `captures/latest_test_session.json`
2. `captures/latest_joint_chain_evidence.json`
3. `captures/latest_uplink_contract_report.json`
4. `captures/latest_acceptance_snapshot.json`
5. `captures/latest_delivery_bundle_report.json`（扩展总闸门，可选）

判定口径：

- 第 1 步看“功能是否跑通”（6 场景是否全 PASS）
- 第 2 步看“证据是否完整”（每场景关键行与字段是否可追溯）
- 第 3 步看“接口是否合规”（状态/事件报文契约是否 PASS）
- 第 4 步看“是否可交付”（`suite_ok/evidence_ready/contract_ok/deliverable_ready`）
- 第 5 步看“总闸门一眼结论”（可选增强）

## 11. 常见失败定位表（计划内）
| 看到的失败现象 | 首查文件 | 典型原因 | 处理建议 |
|---|---|---|---|
| `suite_failed > 0` | `latest_test_session.json` | 场景行为与预期不一致 | 回看 `suite_report.failed_checks` 的 `line`，先修功能链 |
| `evidence_ready=false` | `latest_joint_chain_evidence.json` | 某场景无关键行或字段缺失 | 看 `evidence_items.scenarioX.available/check_ok/reason`，补采样或修步骤 |
| `result=FAIL` 且 `event_latest_unavailable` | `latest_uplink_contract_report.json` | 事件最新条目未写入 | 先跑联合套件再跑契约核对，确保有事件留痕 |
| `contract_ok=false` | `latest_uplink_contract_report.json` | 报文字段缺失/非法值 | 按 `status_failures/event_failures` 对字段修复 |
| `deliverable_ready=false` | `latest_acceptance_snapshot.json` | 三者之一未就绪 | 分别核对 `suite_ok/evidence_ready/contract_ok` 哪个为 false |
| 总闸门失败 | `latest_delivery_bundle_report.json` | 上游任一文件未达标或不一致 | 按 `failures` 列表逐项回溯到对应文件修复 |

```powershell
python tools/uplink_packet_contract_check_报文契约核对.py
python tools/delivery_bundle_readiness_check_交付包就绪核对.py
```

## 12. 时序防误判口径（4.10 补充）

为避免“沿用旧契约 PASS”导致误判，本轮补充以下固定口径：

1. `latest_acceptance_snapshot.json` 新增新鲜度字段  
   - `suite_finished_ms`  
   - `contract_fresh`  
   - `contract_stale_by_ms`

2. 快照内 `contract_ok` 判定升级  
   - 不再只看 `contract_result=PASS`，还必须满足 `contract_fresh=true`。  
   - 即：`contract_ok = PASS and fresh`。

3. 总闸门新增时序一致性校验  
   - 脚本：`tools/delivery_bundle_readiness_check_交付包就绪核对.py`  
   - 新参数：`--max-backward-ms 1500`（默认值）  
   - 核对顺序：  
     - `suite_finished_ms <= contract_checked_ms <= snapshot_updated_ms`（允许小抖动）  
   - 若时间逆序，直接 `result=FAIL` 并给出 failure code。
