# Flytotal v5.1：LD2451 + NodeB + NodeC 多源低空目标感知升级方案

## 1. 项目目标

在不破坏 v1.0 freeze 的前提下，将系统从单一短距雷达感知升级为多源低空目标感知系统。

v5.1 目标不是简单替换雷达，而是构建：

```text
LD2451 长距运动触发
+ LD2450 短距二维角度
+ ESP32-C3 Wi-Fi/BLE 身份链
+ ESP32-P4 独立大屏展示
+ PC Dashboard 兜底
```

最终形成可演示、可测试、可回退、可答辩的低空目标感知与预警闭环。

## 1.1 当前开发节奏

当前实施重点不是一开始同时推进三节点，而是先完成 `NodeA + NodeB` 双节点联调。

阶段顺序锁定为：

```text
Phase 1：NodeA + NodeB 双节点联调
NodeA = ESP32-S3 主控
NodeB = ESP32-C3 Wi-Fi/BLE 监听

Phase 2：LD2451 并联接入
NodeA 同时接收 LD2450、LD2451、NodeB

Phase 3：多源融合与本地告警
完成 LOW / MID / HIGH 融合和 L0-L4 判决

Phase 4：NodeC 独立大屏展示
ESP32-P4 只做展示增强，不进入主控闭环
```

因此，当前优先级是：

1. 先把 `ESP32-C3 NodeB` 的 Wi-Fi/BLE 监听跑通。
2. 再让 `ESP32-S3 NodeA` 稳定接收 NodeB 串口消息。
3. 然后接入 `LD2451`，形成 `LD2450 + LD2451 + NodeB` 三源输入。
4. 最后再接 `ESP32-P4 NodeC` 独立大屏。

这样做的原因是：双节点联调可以先验证身份链和串口协议，不会过早引入 P4、LVGL、MIPI 屏等显示栈风险。

## 2. 核心技术路线

### 2.1 LD2450 保留

`LD2450` 继续作为短距二维坐标雷达。

作用：

- 输出 `x / y` 二维坐标。
- 支撑 `atan2(x, y)` 计算云台 pan 角。
- 保证原有云台跟踪链路不失效。
- 保护 v1.0 freeze。

### 2.2 LD2451 新增

`LD2451` 作为长距运动触发雷达，并联接入，不替代 LD2450。

作用：

- 输出距离、速度、接近/远离状态。
- 只作为 `far_motion_trigger`。
- 不参与云台角度计算。
- 不写入原有 TrackManager 主链路。
- 不承诺对小型无人机 80m 稳定识别，只做真实距离拉测。

### 2.3 NodeB：ESP32-C3 Wi-Fi/BLE 监听

`ESP32-C3` 作为独立 NodeB，负责被动监听无线身份链。

作用：

- Wi-Fi OUI 监听。
- BLE 被动扫描。
- Remote ID / OpenDroneID / DJI 相关特征识别。
- 通过串口向 NodeA 输出身份链事件。
- 只做被动监听，不做干扰、攻击、deauth。

### 2.4 NodeC：ESP32-P4 独立大屏

`ESP32-P4 + 7 寸 MIPI 屏` 作为 NodeC 展示节点。

作用：

- 独立大屏 Dashboard。
- 显示三源状态、风险等级、判决等级、事件日志。
- 只接收 NodeA 的 `DISP` 数据。
- 不反向控制 NodeA。
- P4 崩溃或断电不影响 NodeA 主循环。
- 当前阶段不优先联调，放在双节点联调和 LD2451 接入之后。

## 3. 系统架构

```text
           +----------------------+
           |      LD2450          |
           |  x / y 二维坐标       |
           +----------+-----------+
                      |
                      v
+---------+     +-----+-------------------------------+
| LD2451  | --> | NodeA: ESP32-S3                    |
| range   |     | - 云台控制                          |
| speed   |     | - 融合判断                          |
| approach|     | - 本地告警                          |
+---------+     | - 输出 Dashboard / DISP             |
                +------+-----------------------+------+
                       |                       |
                       v                       v
              +--------+--------+      +-------+--------+
              | NodeB: ESP32-C3 |      | NodeC: ESP32-P4 |
              | Wi-Fi/BLE 监听  |      | 7寸独立大屏     |
              +-----------------+      +----------------+
```

## 4. 融合规则

### 4.1 三源信号

```text
S_LD2450 = LD2450 track confirmed
S_LD2451 = approach && 10m < range < 100m && speed valid
S_RID = RID / OUI / BLE event active within time window
```

### 4.2 融合等级

```text
LOW  = 任意 1 源命中
MID  = 任意 2 源在时间窗内命中
HIGH = 3 源同时命中
```

### 4.3 判决等级

| 等级 | 含义 | 展示 |
|---|---|---|
| L0 | 合法目标，RID 验证通过，雷达一致 | 绿色 |
| L1 | 合法目标，但 RID 缺失或不完整 | 黄色 |
| L2 | 非合作目标，飞行行为暂未异常 | 橙色 |
| L3 | 非合作目标，且存在高速接近/禁飞区等异常 | 红色 |
| L4 | 疑似 RID 仿冒或对抗行为 | 红色闪烁 |

## 6. 执行计划

### Phase 1：NodeA + NodeB 双节点联调

目标：

- NodeB 能独立完成 Wi-Fi/BLE 被动监听。
- NodeB 能通过串口输出统一格式的身份链消息。
- NodeA 能稳定接收 NodeB 消息，不影响原有 LD2450 和云台链路。

验收：

- ESP32-C3 串口每秒至少输出 1 条有效监听消息。
- ESP32-S3 能读取并显示 `rid / source / rssi / status` 等关键字段。
- NodeB 掉线或重启时，NodeA 不阻塞、不崩溃。
- 原 v1.0 六个基线场景不回归。

### Phase 2：LD2451 并联接入

目标：

- LD2451 接入 NodeA。
- Dashboard 能看到长距雷达字段。
- NodeA 同时接收 LD2450、LD2451、NodeB 三类输入。

验收：

- LD2451 能输出 `range / speed / approach`。
- LD2450 的 `x / y` 同时正常。
- NodeB 身份链消息仍能正常进入 NodeA。
- LD2451 不参与 TrackManager 和云台角度计算。

### Phase 3：融合 + 本地告警

目标：

- 实现三源融合规则。
- 实现 L0-L4 判决矩阵。
- 完成本地 RGB / 蜂鸣器 / OLED 告警。
- 准备证据包基础日志。

验收：

- `radar only / RID only / 双源 / 三源` 均可区分。
- 仿冒场景能触发 L4。
- 触发事件时蜂鸣器响、RGB 变色。
- `FusionConfig::EnableBonus` 默认保持 false，避免提前影响基线。

### Phase 4：NodeC 独立大屏展示

目标：

- NodeA 单向发送 `DISP,{json}` 给 NodeC。
- NodeC 显示 NodeA 实时状态。
- P4 大屏用于答辩展示，不参与主控闭环。

验收：

- P4 显示 `Flytotal NodeC OK`。
- NodeC 能显示风险分、雷达状态、身份链状态。
- 拔掉 NodeC 电源，NodeA 主循环不受影响。
- 如果 P4 点屏失败，PC Dashboard 投屏兜底。

### Phase 5：现场首飞 + 阈值调参

目标：

- 完成 10m、30m、50m 首轮真机测试。
- 观察 LD2451、LD2450、NodeB 的真实表现。
- 根据现场数据调整阈值。

验收：

- 至少拿到 10m / 30m / 50m 三档真实 session log。
- 至少一档达到 `fusion=MID`。
- 若 50m 效果差，缩小到 10m / 20m / 30m 保底。

### Phase 6：正式验收 + 答辩素材

目标：

- 完成关键场景测试。
- 生成证据包、视频回放、延迟图、检测指标图。
- 准备答辩素材。

优先场景：

- 10m HIGH
- 30m MID
- 50m LOW
- 仿冒 RID L4
- 双机对抗

验收：

- 至少 3 个事件证据包。
- 至少 1 次 L4 对抗展示。
- Dashboard 屏录不少于 10 分钟。
- NodeC 大屏完成独立展示。
- v1.0 备用 S3 可回退演示。

## 7. 测试计划

每天必须跑：

- v1.0 六个基线场景。
- LD2450 云台链路。
- NodeB 监听输出。
- 本地告警状态。

从 Phase 2 开始增加：

- LD2451 长距字段输出。

从 Phase 4 开始增加：

- NodeC 断电不影响 NodeA。

正式测试必须覆盖：

- 10m / 30m / 50m。
- 有 RID / 无 RID。
- 雷达命中 / 身份链命中 / 三源命中。
- 正常目标 / 仿冒目标。
- 单机 / 双机对抗。

## 8. 风险与兜底

| 风险 | 兜底 |
|---|---|
| LD2451 对无人机距离不稳定 | 改为长距运动触发，不承诺稳定识别 |
| P4 点屏失败 | PC Dashboard 投屏 |
| NodeB Wi-Fi/BLE 共存崩溃 | Wi-Fi 2s / BLE 1s 时分扫描 |
| SoftwareSerial 丢包 | NodeB 优先稳定，NodeC 丢帧可接受 |
| OLED 占用时间过多 | 砍 OLED UI，保留蜂鸣器 + RGB |
| 50m 效果不佳 | 缩小为 10m / 20m / 30m 验收 |
| 户外飞行受限 | 地下停车场/空旷室内做雷达 + RID 验证 |
| P4 不到货 | 不影响主线，PC Dashboard 替代 |

## 9. 明确不做

- 不用 LD2451 替代 LD2450。
- 不让 LD2451 参与云台角度计算。
- 不做 P4 摄像头。
- 不做 P4 触屏交互。
- 不做 NodeC 反向控制 NodeA。
- 不做多 NodeB mesh。
- 不做 75 架次完整矩阵。
- 不承诺 100m 稳定识别小型无人机。
- 不做 Wi-Fi/BLE 干扰或攻击。

## 10. 答辩表述

本项目不是简单堆硬件，而是在保护原有稳定版本的基础上，采用增量式架构升级，实现长距触发、短距定位、无线身份链和独立大屏展示的多源融合预警系统。

核心创新点：

- 长短距雷达协同。
- 无线身份链辅助判断。
- 多源融合风险分级。
- L0-L4 判决矩阵。
- 对抗检测与证据链。
- 独立大屏展示与 PC Dashboard 兜底。
- 可回退的工程化冻结策略。

## 11. 最终结论

v5.1 方案可以执行。

推荐采购：

```text
必买：LD2451、ESP32-C3、天线、备用 S3、供电、线材、告警、记录、测距、支架、防护
推荐：ESP32-P4 + 7寸屏、逻辑分析仪、PD 触发模块、现场维修工具
可选：50m 卷尺、额外录像支架
```

最终验收重点不是 100m 单点成功，而是：

```text
10/30/50m 真实数据
+ 至少一档 fusion=HIGH
+ 至少一档 fusion=MID
+ L4 对抗展示
+ 证据包
+ 独立大屏
+ v1.0 可回退
```
