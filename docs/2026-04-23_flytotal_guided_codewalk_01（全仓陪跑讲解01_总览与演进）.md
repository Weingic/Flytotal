# Flytotal 全仓陪跑讲解 01：总览与演进

这组讲解不是单纯解释“代码写了什么”，而是想同时帮你建立 3 层理解：

1. 代码事实：这段代码现在实际在做什么。
2. 设计理解：为什么要这样拆模块、定状态、设参数。
3. 演进思路：这个功能为什么会在当前阶段自然长出来。

重要说明：

- 文中“代码事实”基于当前仓库代码。
- 文中“设计推断”是根据代码结构、字段命名、调用关系做出的工程理解，不等于作者当时脑中的唯一答案。

---

## 1. 这套系统现在到底是什么

如果只看 `src/main.cpp`，你会感觉这是一个 ESP32 固件工程。  
如果把 `tools/` 和网页一起看，它其实已经长成了一套“板端 + 主机端 + 可视化 + 验收闭环”的联调系统。

可以先把它理解成下面这条主链：

```text
真实雷达 / 模拟输入
    ->
轨迹对象
    ->
风险判断
    ->
云台控制 / 事件生命周期
    ->
串口上行
    ->
主机桥接
    ->
视觉辅助 / 网页展示 / 验收工具 / 证据导出
```

也就是说，这个项目现在已经不只是“驱动一个云台”。

它在尝试解决的是：

- 发现目标以后，如何稳定跟踪。
- 跟踪以后，如何判断风险。
- 判断以后，如何把风险升级成事件。
- 事件以后，如何在主机侧留痕、验收、展示、导出。
- 当单节点讲不清时，如何扩展到双节点接力与证据链。

## 2. 先看代码目录，不要先看单个函数

### 2.1 固件骨架

- `include/AppConfig.h`
  所有关键参数、阈值、超时、默认配置。
- `include/SharedData.h`
  所有任务共享的数据结构和状态枚举。
- `lib/RadarParser`
  把雷达原始字节流转成可用坐标。
- `lib/TrackManager`
  把离散点变成“带生命周期的轨迹对象”。
- `lib/HunterAction`
  负责风险评分、状态升级、触发标志。
- `lib/GimbalPredictor`
  负责把目标位置变成云台角度预测。
- `lib/GimbalController`
  负责云台状态机。
- `lib/Safetylink`
  早期/旁路线的应急通信能力，当前主链里不是核心角色。
- `src/main.cpp`
  系统编排中心，负责把上面所有能力串起来。

### 2.2 主机与网页骨架

- `tools/node_a_serial_bridge_NodeA串口桥接.py`
  把板端串口输出翻译成主机侧 JSON 状态和事件。
- `tools/vision_bridge_视觉桥接.py`
  负责视觉跟踪、抓拍、视觉状态输出。
- `tools/vision_web_server_视觉网页服务.py`
  提供 API，把本地 JSON/CSV/导出文件组织成网页可读数据。
- `tools/vision_dashboard.html`
  面向人看的联调看板。
- `tools/track_injector_轨迹注入器.py`
  在没有真实雷达或需要可重复测试时，向固件注入轨迹和命令。
- `tools/acceptance_*` / `preflight_*` / `consistency_check_*`
  把“联调成功”变成“可检查、可复现、可交付”的流程。

### 2.3 配套但不属于主业务主线的文件

- `include/gimbal_gcs_云台地面站.py`
  早期/辅助型地面站，可实时观察和调 Kp/Kd。
- `tools/rid_broadcast_simulator_身份广播模拟器.py`
  用主机向设备注入身份广播文本，辅助演示 RID 链路。
- `tools/git_integration_*`
  环境同步脚本，不属于业务逻辑，但属于项目协作链路。
- `include/README`、`lib/README`
  PlatformIO 模板说明，不是项目功能代码。

## 3. 这套系统是怎么一步步长出来的

这部分很重要，因为你现在“看不懂”的根源，往往不是单个函数难，而是没有看到项目为什么会越长越大。

### 阶段 1：先解决“目标来了怎么办”

最早的问题一定很朴素：

- 雷达给了数据，怎么读？
- 目标出现了，云台怎么动？

所以最早长出来的就是：

- `RadarParser`
- `TrackManager`
- `GimbalPredictor`
- `GimbalController`

这一阶段的核心目标不是“判断危险”，而是“能不能稳定发现并跟着目标走”。

设计灵感：

- 先把底层感知链打通。
- 先让系统动起来，哪怕还不聪明。

### 阶段 2：只跟踪还不够，要判断“它危险不危险”

当系统能跟目标走之后，就会自然暴露一个问题：

- 不是所有目标都一样。
- 不能所有出现的目标都当成高风险。

所以接着就长出了：

- `HunterAction`
- 风险分数组成
- `HunterState`
- `trigger_alert` / `trigger_capture` / `trigger_guardian`

设计灵感：

- 用连续分数表达“风险程度”，比单纯真假判断更细。
- 再用状态机把连续分数收敛成可解释的阶段。

### 阶段 3：只判断还不够，要把风险变成事件

当系统已经会判断风险后，下一个问题就是：

- 风险是一瞬间的，还是一个有开始、有保持、有关闭过程的东西？

于是 `main.cpp` 里长出了大量事件管理结构：

- `EventContext`
- `EventLifecycleState`
- `RuntimeEventStatus`
- `EventPolicySnapshot`
- `EventObject`

设计灵感：

- 事件不是一个布尔值，而是一个“生命周期对象”。
- 只有这样，后面才可能做导出、抓拍、闭环、回放。

### 阶段 4：只留痕还不够，要知道它是否合法

风险系统做出来以后，实际工程里马上会遇到：

- 目标可能是合法合作目标。
- 不能因为看到目标就永远升级风险。

于是系统又长出：

- `RidStatus`
- `WhitelistStatus`
- `RidIdentityPacket`
- 白名单判定
- RID 接收超时、合法保持、重确认窗口

设计灵感：

- 不把目标简单分成“有/无”。
- 引入身份链，让系统从“看到目标”走向“理解目标身份”。

### 阶段 5：单节点故事不够，要能讲接力

当单节点能完成发现、判断、事件之后，如果要继续演示更完整的场景，就会出现：

- 目标从 A1 区域飞到 A2 区域怎么办？
- “谁先发现、谁接手、谁负责”怎么讲清楚？

所以 `main.cpp` 和主机侧又长出了：

- `NodeRuntimeCache`
- `NodeRiskContext`
- `handoff_from` / `handoff_to`
- `prev_node_id`
- `continuity_hint`
- `source_node`

设计灵感：

- 双节点的重点不是算法炫技，而是“责任交接可解释”。
- 所以这里优先保存显式字段，而不是把接力关系藏在隐式逻辑里。

### 阶段 6：只跑通代码还不够，要可观察、可验收、可交付

这是很多初学者最容易忽略的一层。

当固件已经很能干之后，项目并不会停。因为真实工程里接着会问：

- 你怎么证明系统在工作？
- 你怎么让别人看到结果？
- 你怎么做稳定性检查？
- 你怎么导出证据？

这就是为什么工具链会迅速膨胀：

- 串口桥接
- 视觉桥接
- 网页服务
- 看板
- 轨迹注入
- 预检
- 自动验收
- 一致性核对
- 证据链哈希
- 误报率统计
- E2E 延迟统计

设计灵感：

- 嵌入式项目不能只有板端代码。
- 真正能交付的系统，一定要补齐“外部可观察层”和“验证闭环层”。

## 4. 为什么这套系统喜欢用状态机、阈值、保持时间

你会在仓库里看到很多：

- `*_Threshold`
- `*_HoldMs`
- `*_TimeoutMs`
- `cooldown`
- `reopen`

这不是“写复杂了”，而是这类系统天然需要对抗 4 个问题：

1. 噪声
   雷达、视觉、串口都可能抖。
2. 误触发
   目标刚出现一帧，不一定就该进入强动作。
3. 来回跳变
   如果没有 hold/cooldown，状态会疯狂开关。
4. 解释性
   比赛、联调、验收时，必须讲清“为什么现在是这个状态”。

设计推断：

- 状态机负责把行为离散化。
- 阈值负责把连续量切出边界。
- 保持时间负责抗抖。
- 冷却时间负责防止事件重复开合。

这是一种很典型的工程型设计，不追求最学术，但追求稳定、能讲清、能验收。

## 5. 这套仓库的阅读顺序

如果你基础薄弱，不要上来硬读 `src/main.cpp` 近 5000 行。

推荐顺序如下：

1. `include/AppConfig.h`
   先认识“系统想控制哪些参数”。
2. `include/SharedData.h`
   再认识“系统内部到底在传什么状态”。
3. `lib/RadarParser`
   看原始输入如何变成坐标。
4. `lib/TrackManager`
   看坐标如何变成轨迹。
5. `lib/HunterAction`
   看轨迹如何变成风险。
6. `lib/GimbalPredictor` + `lib/GimbalController`
   看风险之外，控制链如何工作。
7. `src/main.cpp`
   最后看系统编排层。
8. `tools/node_a_serial_bridge_NodeA串口桥接.py`
   看板端输出如何被主机接住。
9. `tools/vision_bridge_视觉桥接.py`
   看视觉如何补进主链。
10. `tools/vision_web_server_视觉网页服务.py` + `tools/vision_dashboard.html`
   看系统如何被“人”观察。
11. 其余测试和验收工具
   看项目如何变得可验证。

## 6. 后续两讲会怎么展开

### 第二讲：固件主链

会覆盖这些文件：

- `include/AppConfig.h`
- `include/SharedData.h`
- `include/gimbal_gcs_云台地面站.py`
- `lib/RadarParser/*`
- `lib/TrackManager/*`
- `lib/HunterAction/*`
- `lib/GimbalPredictor/*`
- `lib/GimbalController/*`
- `lib/Safetylink/*`
- `src/main.cpp`

重点回答：

- 从一帧输入到一次上行，板端到底怎么接力。
- 为什么 `main.cpp` 会长成现在这样。
- 哪些能力是“主链核心”，哪些是“编排层补强”。

### 第三讲：工具链与网页

会覆盖这些文件：

- `tools/*.py`
- `tools/vision_dashboard.html`
- `tools/track_scenarios.json`
- `tools/git_integration_*`

重点回答：

- 为什么这些主机脚本不是“边角料”，而是系统的一部分。
- 这些工具分别在模拟、桥接、展示、验收、交付哪一环起作用。
- 为什么系统最终会自然长出这么多辅助脚本。

## 7. 这一讲你先记住什么

先记住 4 句话：

1. 这个项目已经不是单纯固件，而是“固件 + 工具链 + 网页 + 验收闭环”的组合系统。
2. 它的主干逻辑很稳定，本质就是“感知 -> 判断 -> 输出 -> 留痕 -> 展示”。
3. 复杂度主要不是来自单个算法，而是来自系统能力一层层叠加。
4. 你后面读代码时，要始终问自己：这段是在解决“发现、判断、控制、留痕、展示、验收”里的哪一个问题。
