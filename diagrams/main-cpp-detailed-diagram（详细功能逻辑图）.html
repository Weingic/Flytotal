`main.cpp` 详细功能逻辑图

这张图专门解释 src/main.cpp。Arduino/ESP32 工程没有传统意义上的 main() 供你直接写业务，这里真正的主入口是 setup() 和 loop()。而你的业务主体又被拆成 3 个 FreeRTOS 任务：RadarTask、TrackingTask、CloudTask。
Entry Model
1. `setup()` / `loop()` 等价于主函数入口
如果你要找“主函数到底干了什么”，就先看 `setup()`：它做初始化、打印启动信息、创建任务。`loop()` 本身不承载业务，只是删除自己。
Serial.begin()

参数：AppSerialConfig::MonitorBaudRate
作用：打开 USB 监视串口

→
delay()

参数：AppSerialConfig::StartupDelayMs
作用：预留启动等待时间

→
resetSummaryStats()
resetHandoverStatus()
resetLastEventSnapshot()
resetRuntimeEventStatus()

作用：清空统计、handover、事件快照和运行态事件

→
xTaskCreatePinnedToCore()

创建 3 个核心任务
RadarTask / TrackingTask / CloudTask

→
loop() -> vTaskDelete(NULL)

自身退出，主业务完全交给任务执行

Task Layout
2. 三大任务的职责
RadarTask

读雷达口 / 读取模拟轨迹
维护 globalData.radar_track

→
TrackingTask

处理主机命令
计算 hunter 风险
更新 vision / gimbal / trigger 状态

→
CloudTask

把统一状态发布成
UPLINK,HB / UPLINK,TRACK / UPLINK,EVENT

RadarTask 启动参数

  • 栈：4096
  • 优先级：1
  • 核心：0
  • 主要配置：RadarBaudRate、RadarRxPin、RadarTxPin、RadarConfig::PollDelayMs

TrackingTask 启动参数

  • 栈：6144
  • 优先级：2
  • 核心：1
  • 主要配置：TrackingConfig::LoopDelayMs、TrackingConfig::LostRecoveryTimeoutMs、Servo/Gimbal/Hunter 配置

CloudTask 启动参数

  • 栈：4096
  • 优先级：1
  • 核心：1
  • 主要配置：CloudConfig::HeartbeatMs、CloudConfig::EventReportMs
Radar Task
3. `RadarTask()` 详细流程
这块负责“轨迹来源”。如果有模拟轨迹在生效，优先使用模拟轨迹；否则就走真实雷达串口输入。
Serial1.begin()

参数：RadarBaudRate / RadarRxPin / RadarTxPin
打开雷达硬串口

→
getSimTrackSnapshot()

查看模拟轨迹是否启用
以及是否仍在有效时间内

→
myRadar.feedByte()

逐字节喂给 RadarParser
解析出 x / y

→
applyTrackMeasurement()

更新轨迹测量值
模拟轨迹优先级高于真实雷达输入

→
myTrackManager.refresh()

更新 active / confirmed / lost 等轨迹状态
写回 globalData.radar_track 和 is_locked

Tracking Task
4. `TrackingTask()` 详细流程
这块是整条主控链的核心决策器：主机命令、风险判断、视觉状态推导、舵机控制、调试输出都发生在这里。
pollHostCommands()

进入 handleHostCommand()

→
processServoDiagnostic()

推进舵机诊断流程

→
myHunter.update()

核心风险判断器
输出：state / risk_score / trigger_alert / trigger_capture / trigger_guardian

→
推导视觉状态

根据轨迹 active / confirmed / trigger_capture
推导 VISION_IDLE / SEARCHING / LOCKED / LOST

→
回写 globalData

写 hunter / risk / vision / capture_ready / trigger_flags / event 状态

→
myGimbalController.update()

算出 gimbal_state 和目标 pan / tilt

→
舵机输出 + 本地调试

控制舵机角度
必要时打印 GIMBAL / DATA / STATE

主要输入

  • track_snapshot：来自 RadarTask 的统一轨迹
  • rid_snapshot：当前 RID 状态
  • 主机命令：来自 USB 串口

主要输出

  • hunter_state
  • risk_score 与风险细分分数
  • vision_state / vision_locked / capture_ready
  • gimbal_state
  • trigger_flags

关键配置参数

  • TrackingConfig::LoopDelayMs：主循环周期
  • TrackingConfig::LostRecoveryTimeoutMs：视觉 LOST 保持时间
  • ServoConfig::PanPin / TiltPin / PulseMinUs / PulseMaxUs
  • GimbalConfig::PredictorKp / PredictorKd
  • HunterConfig::*Threshold：风险阈值
Cloud Task
5. `CloudTask()` 详细流程
这块把设备内部统一状态发布成串口上行文本，也是 Python 侧桥接脚本看到的核心输出来源。
读取 globalData

获得当前统一系统快照

→
emitCloudHeartbeat()

周期性输出 UPLINK,HB
参数：CloudConfig::HeartbeatMs

→
事件上下文管理

判断 EVENT 是否打开/关闭
维护 EventContext

→
handover 请求处理

如果有 pending request，就判断是 ignored 还是 emitted

→
emitCloudTrack()

周期性输出 UPLINK,TRACK
参数：CloudConfig::EventReportMs

→
emitCloudEvent()

在 TRACK_CHANGED / TRACK_ACTIVE / TRACK_LOST / HUNTER_STATE / RID_STATE 等变化时输出 UPLINK,EVENT

主要参数

  • CloudConfig::HeartbeatMs：心跳间隔
  • CloudConfig::EventReportMs：轨迹报告间隔
  • uplinkOutput.enabled：是否允许上行
  • EventContext：当前事件 ID、sequence、track_id

输出到哪里

  • UPLINK,HB：心跳快照
  • UPLINK,TRACK：轨迹快照
  • UPLINK,EVENT：事件变化
  • 这些输出会被 node_a_serial_bridge 读取、解析并转成 JSON
Host Command Entry
6. 主机命令是怎么进入设备端的
Python 工具给 ESP32 发的命令，本质上都是通过 `pollHostCommands()` 这条入口进入设备端逻辑。
Serial.available()

从 USB 串口取字节

→
commandBuffer

逐字节组一整行命令

→
handleHostCommand(commandBuffer)

按命令内容进入不同处理分支

→
影响运行态

可能修改模拟轨迹、RID、KP/KD、测试模式、舵机模式、调试输出、上行输出

来自 `track_injector` 的命令

  • STATUS
  • RID,...
  • TRACK,x,y
  • TRACK,CLEAR
  • RESET

来自单板测试脚本的命令

  • HELP
  • SELFTEST
  • KP,... / KD,...
  • 以及 RID、TRACK、RESET 这些测试命令

会影响的模块

  • 模拟轨迹输入
  • RID 状态
  • 预测器参数 KP/KD
  • 舵机诊断 / 手动模式 / 安全模式
  • 上行状态和调试输出
