`main.cpp` 小白版工人分工图

如果你一看到 RadarTask、TrackingTask、CloudTask 就头大， 可以先别把它们当“任务”看，而是把它们当成设备里的 3 个工人。一个工人负责“看目标”，一个工人负责“做判断和控制”，一个工人负责“往外汇报”。
ENTRY
1. 设备一上电，先发生什么
Arduino/ESP32 里没有你平时常见的业务 `main()`。这里你可以把 setup() 理解成“开工前准备”，把 loop() 理解成“主线程自己退场”。
setup()

先打开监视串口，等待一点时间，让设备稳定启动。

→
清空旧状态

把统计、handover、last event、runtime event 这些运行时状态先重置掉。

→
创建 3 个工人

RadarTask、TrackingTask、CloudTask 开始工作。

→
loop()

主线程不再自己干业务，真正的事情都交给这 3 个工人。

THREE WORKERS
2. 设备里的 3 个工人分别干什么
工人 1：RadarTask

负责“看见目标在哪里”。
来源可以是真实雷达，也可以是电脑发来的模拟轨迹。

→
工人 2：TrackingTask

负责“想一想现在该怎么办”。
包括风险判断、视觉状态推导、云台控制、处理主机命令。

→
工人 3：CloudTask

负责“把结果往外说”。
把当前状态、轨迹、事件打包成串口上行文本。

WORKER 1
3. 工人 1：`RadarTask` 做什么
你可以把它理解成“目标位置采集员”。它的工作不是下判断，而是把“目标现在在哪”这件事弄清楚。
看来源

先看现在有没有“模拟轨迹”。
如果电脑正在发 TRACK,x,y，它会优先用这个。

→
看真实雷达

如果没有有效模拟轨迹，就继续读真实雷达串口数据。

→
整理成统一轨迹

把位置送进 TrackManager，判断是不是 active、是不是 confirmed、是不是丢了。

→
写到共享区

把结果写进 globalData.radar_track，后面的工人就能直接用。

它最关心的输入

  • 真实雷达字节流。
  • 模拟轨迹状态。

它最重要的输出

  • 当前目标坐标。
  • 轨迹是否 active。
  • 轨迹是否 confirmed。
WORKER 2
4. 工人 2：`TrackingTask` 做什么
这是整个设备最核心的工人。你可以把它理解成“决策员 + 控制员”。前面的工人只负责告诉它目标在哪，它负责决定接下来该怎么做。
先收命令

每轮都先检查电脑有没有发新命令。
例如：TRACK、RID、STATUS、RESET。

→
读共享轨迹

拿到工人 1 写好的轨迹快照和当前 RID 状态。

→
做风险判断

调用 myHunter.update()，决定当前风险分数、风险等级，以及要不要 alert / capture / guardian。

→
推导视觉状态

根据轨迹活跃情况和触发状态，推导 VISION_IDLE / SEARCHING / LOCKED / LOST。

→
算云台动作

调用 myGimbalController.update()，决定云台现在该往哪里转。

→
写回共享区

把 hunter、risk、vision、capture_ready、gimbal_state 等结果都写回去。

它最关心的输入

  • 轨迹是否存在。
  • 轨迹坐标和确认状态。
  • RID 状态。
  • 主机命令。

它做出的主要判断

  • 风险高不高。
  • 是否需要触发事件。
  • 视觉状态是什么。
  • 云台该怎么动。

你最常看到的结果

  • risk_score
  • risk_level
  • vision_state
  • gimbal_state
WORKER 3
5. 工人 3：`CloudTask` 做什么
前两个工人都更像在设备内部干活，这个工人负责“把设备内部结果说出去”。所以它最像播报员。
读共享区

拿工人 1 和工人 2 已经整理好的统一状态。

→
发心跳

定期发 UPLINK,HB，相当于“我还活着”。

→
发轨迹快照

定期发 UPLINK,TRACK，告诉外面当前轨迹情况。

→
发事件

当轨迹变了、事件开了、RID 变了、handover 状态变了时，发 UPLINK,EVENT。

→
外部脚本接收

node_a_serial_bridge 就是靠它这些输出，才能整理成 JSON。

HOST COMMANDS
6. 电脑发命令进来后，为什么会影响设备
因为这些命令不是“记在日志里就结束了”，而是真的会改设备的内部状态。

TRACK,x,y

  • 给设备一个模拟目标坐标。
  • 工人 1 会优先把它当成当前目标。

RID,...

  • 改当前 RID 模拟状态。
  • 会影响风险判断和事件结果。

STATUS / SELFTEST / SUMMARY

  • 不是改状态，而是让设备把当前状态打印出来。

KP / KD

  • 改预测器参数。
  • 会影响云台控制感觉。

RESET

  • 清理很多运行时状态。
  • 方便开始下一轮干净测试。

其它命令

  • SAFE、DEBUG、UPLINK 等，会改变诊断和输出行为。
REMEMBER
7. 你只要先记住这 4 句
1
RadarTask 只负责把“目标在哪”这件事弄清楚。
2
TrackingTask 才是设备里真正做判断和控制的核心。
3
CloudTask 不做核心判断，它更像播报员。
4
电脑发进来的命令，最终几乎都会通过 handleHostCommand() 改变这 3 个工人的后续行为。
