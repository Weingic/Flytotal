`node_a_serial_bridge` 详细功能逻辑图

这张图对应 tools/node_a_serial_bridge_NodeA串口桥接.py。它的核心定位不是“控制设备”，而是 “把 Node A 串口吐出来的文本消息，转换成统一 JSON 状态、事件历史和测试结论”。它还会周期性给设备发 STATUS、SELFTEST、SUMMARY，因此它是“轮询器 + 解析器 + 结果汇总器”三合一。

它把设备侧文本协议变成上层脚本和网页更容易消费的结构化状态。

Main Function
2. `main()` 的完整流程
`main()` 做的事情很多，但可以理解成 4 步：准备初始文件、连接串口、循环轮询、按行解析并落盘。
require_pyserial()

确保装了 pyserial
否则无法读串口

→
argparse

解析串口参数、轮询参数、结果文件参数、离线超时参数

→
初始化状态

build_initial_status()
build_initial_test_result()
先把 4 份 JSON 写成“空白初值”

→
打开串口

失败就 return 1
成功后清空缓冲区

→
等待 boot-wait

给设备启动缓冲时间

→
轮询发送命令

用 maybe_send_command()
按时间间隔发送 STATUS、SELFTEST、SUMMARY

→
读取串口行

每来一行都进入事件分支、状态分支、测试结果分支

Main Parameters
3. `main()` 参数图

串口连接参数

  • --port：必填，Node A 所在串口。
  • --baud：默认 115200。
  • --timeout：串口单次读取超时，默认 0.2 秒。
  • --boot-wait：打开串口后先等多久再开始轮询，默认 1.0 秒。
  • --echo：是否把原始串口文本直接打印到终端。

轮询节奏参数

  • --status-interval：自动发送 STATUS 的周期，默认 2.0 秒。
  • --selftest-interval：自动发送 SELFTEST 的周期，默认 6.0 秒。
  • --summary-interval：自动发送 SUMMARY 的周期，默认 0，表示默认关闭。
  • --offline-timeout：超过多久没有有效状态更新就判离线，默认 5.0 秒。

文件与历史参数

  • --output-file：最新统一状态 JSON。
  • --events-file：最近事件历史 JSON。
  • --session-file：读取 track_injector 写出的当前 session。
  • --result-file：本轮最新测试结论 JSON。
  • --result-history-file：历史测试结论列表。
  • --result-history-limit：保留最近多少次结论，默认 16。
  • --events-limit：保留最近多少条事件，默认 12。
  • --session-log-dir：统一时间线目录。
Parsing Pipeline
4. 一行串口文本是怎么变成统一状态的
这一段是整份脚本最重要的“协议桥接逻辑”。设备侧字段名可能不稳定或来源不同，这里会统一成固定 JSON 字段。
parse_prefix_and_fields()

拆出消息前缀和字段字典
例如前缀可能是 STATUS、SELFTEST、SUMMARY、UPLINK,EVENT

→
coerce_value()

把字符串转成 int / float / 原始字符串

→
normalize_fields()

根据 alias_map 做字段别名统一
例如 track → track_id
active → track_active
rid → rid_status
x/y → x_mm/y_mm

→
update_status_from_line()

只处理被监控的前缀
把统一字段 merge 进当前状态字典
同时记住 last_raw_line

→
refresh_runtime_flags()

根据 last_update_ms 计算 stale_age_ms 和 online

Event Branch
6. 事件分支怎么工作
不是每一条串口信息都会进入事件历史。只有被认定为事件前缀的消息，才会生成事件记录。
build_event_record()

只处理事件前缀
提取 event_id、reason、event_level、event_status、track_id、risk_score

→
append_event_record()

用 source_type + timestamp + event_id + reason + track_id 做轻量去重
只检查最近几条，避免刷屏

→
write_event_history_json()

写入 latest_node_events.json
包含 count、records、latest

→
append_session_timeline_event()

再同步写入 session timeline
事件类型是 node_event

Status Branch
7. 状态分支怎么工作
extract_state_signature()

只提取关键状态签名
main_state、risk_level、event_id、track_active、handover_last_result

→
update_status_from_line()

如果这一行属于监控前缀，就刷新当前统一状态

→
write_status_json()

写入 latest_node_status.json

→
状态变化事件

如果当前签名和上一次不同
追加 node_status_changed 到 timeline

Result Logic
8. 它怎么判断 PASS / WARN / FAIL
这里不是简单看一条串口，而是结合 session 生命周期、SUMMARY 计数器增量、风险等级、事件是否发生来综合判断。
update_test_result_monitor()

当 session 进入 RUNNING / SETTLING / DONE 时初始化监视器
保存 baseline summary
并开始跟踪 saw_track_active、saw_event、saw_handover、max_risk_score

→
build_test_result_payload()

计算当前 summary 相对 baseline 的增量
例如 track_active_delta、event_opened_delta、handover_ignored_delta

→
判定规则

没见到轨迹: FAIL
有 handover_ignored: WARN
风险高、发生事件、发生交接: WARN
其余: PASS

→
正式落盘

只在 session 到达 DONE 且还未 finalize 时写结果
并写入历史文件与 timeline

Output Files
9. 最终会产出哪些文件

状态与事件类

  • latest_node_status.json：统一后的当前节点状态。
  • latest_node_events.json：最近事件历史，包含最新事件索引。
  • session_logs/*.jsonl：时间线里会补充 node_event、node_status_changed。

测试结论类

  • latest_test_result.json：本轮最终结论。
  • latest_test_results.json：最近若干轮结论历史。
  • 结果里会带上场景名、RID 模式、最终主状态、最终风险等级、是否发生事件/交接、KP/KD、心跳周期等信息。
