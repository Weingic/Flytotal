`vision_bridge` 详细功能逻辑图

这张图对应 tools/vision_bridge_视觉桥接.py。它的定位是“最小视觉锁定与抓拍桥接器”： 打开摄像头或视频源，允许你手动框选 ROI，使用 OpenCV tracker 持续跟踪目标，并把视觉状态、抓拍记录、最新状态 JSON、 session timeline 一起维护起来。
Role
1. 这个脚本负责什么
它不是做检测模型推理，而是做“人工选目标 + 跟踪 + 抓拍 + 状态桥接”。

Main Function
2. `main()` 的完整流程
`main()` 的核心是“初始化一次，然后进入逐帧循环”。所有状态变化、抓拍、按键控制都在这个循环里完成。
require_opencv()

检查是否安装 OpenCV
缺失就退出并提示安装

→
argparse

解析视频源、tracker 类型、尺寸、日志、抓拍、状态文件参数

→
resolve_path()

把目录和文件路径转绝对路径
包括 capture-dir、status-file、session-file

→
open_source()

打开摄像头索引或视频文件
失败就 return 1

→
初始化运行时变量

tracker、current_bbox、current_state、capture_index、last_capture_record 等

→
进入逐帧循环

读帧、更新 tracker、写状态、自动抓拍、显示 overlay、处理按键

Main Parameters
3. `main()` 参数图

视频输入参数

  • --source：默认 0，可填摄像头索引或视频文件路径。
  • --tracker：可选 csrt 或 kcf，默认 csrt。
  • --width / --height：请求摄像头采集尺寸，默认 1280x720。

状态输出参数

  • --log-interval：锁定状态下重复终端打印的最小间隔，默认 1.0 秒。
  • --log-file：可选，连续视觉状态 CSV。
  • --status-file：最新视觉状态 JSON。
  • --status-write-interval：状态未变化时，最小 JSON 重写间隔，默认 0.25 秒。

抓拍与 session 参数

  • --capture-dir：图片保存目录，默认 captures。
  • --capture-log-file：抓拍元数据 CSV，默认 <capture-dir>/capture_records.csv。
  • --capture-cooldown：自动抓拍冷却时间，默认 2.0 秒。
  • --session-file：当前测试 session JSON。
  • --session-log-dir：timeline 目录。
  • --event-id：附加到抓拍文件名里的事件标签。
Core Types
4. 这份代码里最重要的数据结构

`VisionSnapshot`

  • 描述“某一帧”的视觉状态。
  • 包含 frame_index、vision_state、vision_locked、bbox、center、tracker_name。

`CaptureRecord`

  • 描述“某一次抓拍”的记录。
  • 额外保存 event_id、file_path、capture_reason。

`CsvVisionLogger`

  • 连续状态日志。
  • 每次写一帧的基础视觉状态行。

`CaptureMetadataLogger`

  • 抓拍元数据日志。
  • 每次抓拍成功才写一行。
Frame Loop
5. 每一帧内部到底发生了什么
下面这一段几乎就是你理解整份文件的核心。逐帧循环里依次做“读图、更新 tracker、构造快照、判断打印、判断写状态、判断自动抓拍、画 overlay、处理键盘”。
cap.read()

从视频源读取一帧
失败就结束程序

→
tracker.update(frame)

如果 tracker 存在，就尝试更新 bbox
成功且宽高大于 0: VISION_LOCKED
失败或 bbox 无效: tracker 清空并转 VISION_LOST

→
build_snapshot()

把当前状态打包成 VisionSnapshot
自动计算中心点 (center_x, center_y)

→
vision_signature()

如果视觉签名发生变化，就写入 session timeline 的 vision_status_changed

→
print_snapshot()

状态变化时立即打印
锁定期间也会按 log-interval 周期重复打印

→
write_latest_status_json()

状态变化或到达 status-write-interval 时刷新 latest_status.json

→
draw_overlay() + imshow()

画框、中心点、状态文本、帮助按键提示并显示窗口

State Machine
6. 状态机图

`VISION_IDLE`

  • 初始状态。
  • 没有 tracker，没锁定目标。
  • 按 s 或空格后准备选框。

`VISION_SEARCHING`

  • 进入 ROI 选择流程。
  • 执行 select_target_roi()。
  • 框选成功则转锁定，否则回空闲。

`VISION_LOCKED`

  • tracker 初始化成功并持续更新成功。
  • 可自动抓拍，也允许手动抓拍。

`VISION_LOST`

  • 之前锁定过，但 tracker 更新失败。
  • tracker 会被清空。
  • 之后可重新选框回到搜索流程。
Capture Logic
7. 自动抓拍和手动抓拍是怎么触发的
自动抓拍触发条件

is_new_lock 为真
也就是当前帧锁定且上一状态不是 VISION_LOCKED

→
冷却判定

now - last_auto_capture_time >= capture_cooldown
防止刚锁上就疯狂连续抓拍

→
capture_if_allowed()

内部调用 save_capture()
真正落盘 JPG，并生成 CaptureRecord

→
记录副作用

打印 CAPTURE,...
写抓拍 CSV
更新 last_capture_record

→
时间线与状态

写 capture_saved 到 session timeline
并立即刷新 latest_status.json

Key Bindings
8. 键盘控制图

选择与重置

  • s / 空格：进入 VISION_SEARCHING，弹出 ROI 选框。
  • r：重置 tracker，清空 bbox，状态回到 VISION_IDLE。

抓拍与帮助

  • c：仅在 VISION_LOCKED 时允许手动抓拍。
  • h：切换帮助文字显示。

退出

  • q 或 ESC：正常退出。
  • 退出时会释放视频源、关闭窗口、关闭 CSV 记录器。
Output Files
9. 最终会生成哪些文件

实时状态类

  • latest_status.json：最新帧状态。
  • 字段包括 vision_state、vision_locked、bbox_x/y/w/h、center_x/y、last_capture_reason、last_capture_file。
  • 可被本地网页服务或其他脚本直接读取。

抓拍与日志类

  • capture_records.csv：每次抓拍的元数据。
  • captures/*.jpg：真正保存的抓拍图片。
  • log-file 对应 CSV：连续帧状态日志，可选。
  • session_logs/*.jsonl：额外补充 vision_status_changed 和 capture_saved。
