Flytotal 小白版总逻辑图

这不是“高手速读版”，而是“第一次看也能跟上”的版本。你可以把整个项目想成一条流水线： 先有人把测试目标喂给设备，再由设备自己处理，再把结果吐回来，最后再由 Python 脚本整理成你能看的 JSON、图片和日志。

为什么你之前会看不懂

因为原来的图更像“给已经懂串口、任务、tracker、session 的人看的工程图”。如果你还不熟这些词，光看函数名很难把一整条链串起来。

这张图怎么读

先只看“5 个角色”，再看“一轮测试从头到尾怎么走”，最后再看“参数到底是什么意思”。不要一开始就钻到某个函数里。

ONE FULL TEST
2. 一轮测试从开始到结束怎么走
如果你只想知道“我一按运行，到底发生了什么”，就看这一段。
第 1 步

track_injector 读取一个测试场景。
比如：先发点 (-600, 1800)，再发点 (-450, 1800)……

→
第 2 步

它通过串口给设备发命令。
最常见的是 TRACK,x,y。

→
第 3 步

main.cpp 收到命令后，更新内部“目标位置、风险状态、事件状态、云台状态”。

→
第 4 步

设备再把自己的状态打印出来。
比如：STATUS,...、SELFTEST,...、UPLINK,EVENT,...。

→
第 5 步

node_a_serial_bridge 把这些文本整理成 JSON。
这样网页和其他脚本就不用自己解析文本了。

第 6 步

如果同时开着 vision_bridge，它会看摄像头画面、框选目标、锁定并抓拍。

→
第 7 步

最终你会得到 4 类结果：
设备状态 JSON、测试结论 JSON、抓拍图片、统一时间线日志。

WHAT EACH FILE DOES
4. 每个关键文件到底是干什么的

`track_injector_轨迹注入器.py`

  • 读取测试场景。
  • 按顺序发 TRACK 点。
  • 写当前测试进度到 latest_test_session.json。

`src/main.cpp`

  • 设备端主程序。
  • 处理主机命令。
  • 计算风险、事件、云台状态。
  • 把状态打印成上行文本。

`node_a_serial_bridge.py`

  • 自动轮询设备状态。
  • 解析设备文本。
  • 输出统一 JSON。
  • 在测试结束时给出 PASS / WARN / FAIL。

`vision_bridge.py`

  • 打开摄像头或视频。
  • 人工选框后跟踪目标。
  • 自动或手动抓拍。
  • 输出视觉状态 JSON 和图片。
BEGINNER PARAMS
5. 你最常见的参数，直接翻成人话

`track_injector` 参数

  • --port：连的是哪个串口。
  • --baud：串口传输速度。通常保持默认就行。
  • --scenario：跑哪个测试场景。
  • --interval：两次发点之间隔多久。
  • --hold-repeat：同一个点重复发几次。越大越像“停留”。
  • --settle：发完清轨命令后再等多久。
  • --rid：先给设备一个 RID 状态。

`node_a_serial_bridge` 参数

  • --status-interval：多久问一次设备“你现在状态怎样”。
  • --selftest-interval：多久问一次设备“你的配置和健康状态怎样”。
  • --summary-interval：多久问一次设备“统计汇总怎样”。
  • --offline-timeout：多久没更新就认为设备离线。
  • --events-limit：最多保留多少条事件。

`vision_bridge` 参数

  • --source：摄像头编号或视频文件路径。
  • --tracker：跟踪器类型，简单理解成“框选后怎么继续跟住目标”。
  • --capture-cooldown：两次自动抓拍之间至少隔多久。
  • --event-id：给抓拍文件名加一个标签。
  • --status-file：把最新视觉状态写到哪个 JSON 文件。
TERMS
6. 你现在最容易卡住的名词解释

串口 `port`

  • 就是电脑和设备通信的“门牌号”。
  • Windows 下常见是 COM3、COM4 这种。

波特率 `baud`

  • 可以简单理解成“串口说话速度”。
  • 两边必须一致，不一致就会乱码。

`session`

  • 就是“当前这轮测试”的状态记录。
  • 例如：正在跑哪个场景、跑到第几个点、现在是 RUNNING 还是 DONE。

`tracker`

  • 不是检测器。
  • 它更像“你先框一个目标，后面程序尽量继续跟住这个框”。
IF YOU ONLY REMEMBER 3 THINGS
7. 你先记住这 3 件事就够了
1
track_injector 是“喂测试数据的人”，不是处理结果的人。
2
main.cpp 才是设备端真正做判断、做状态变化的地方。
3
node_a_serial_bridge 和 vision_bridge 都更像“记录员”，一个记设备状态，一个记视觉状态。
